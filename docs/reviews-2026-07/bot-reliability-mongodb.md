TOP FINDINGS — MongoDB connection + data-layer robustness (Butler / Jester / Oracle)

Live-DB evidence gathered via MongoDB MCP: `Database` is tiny — 18,148 docs / 22 MB total, `lunari_transactions` = 6,137 docs (~4 months of logging), `nemesis` = 62 docs, `game_wins` = 68 docs, 117 indexes exist DB-wide. This calibrates several "gameplan" fears downward (details in findings 7–8).

---

**1. Butler + Oracle: DB init failure at boot → permanent zombie bot (PM2 thinks it's healthy)** — Severity: HIGH, Effort: S
- Evidence: `C:\Users\Admin\Desktop\Luna Bot\LunaButlerMain\index.ts:188-192` — `botInitialized = true` is latched BEFORE `await initDatabase()`. `initDatabase` (`util\helpers\database_helper.ts:19-33`) throws after 5 attempts (~30s). The throw becomes an unhandled rejection which `index.ts:143-149` only logs, and `process.exit(1)` is commented out at `index.ts:162`. Oracle identical: `LunaOracle\index.ts:192-197` latches `botInitialized`, and the catch at `index.ts:372-374` just logs `[Oracle] Init error`.
- Failure scenario: Atlas blip/DNS failure during the daily 4 AM cron restart → init throws once → bot stays connected to Discord with ALL managers undefined (Butler: every command throws `TypeError: Cannot read properties of undefined`; Oracle: voice handlers never registered, join-to-create silently dead). PM2 sees a live process, no restart, broken until a human notices.
- Fix: on init failure, `process.exit(1)` and let PM2's restart/backoff do the retrying (this is the existing self-heal channel — the 354/237 restart counts prove PM2 recovery works). Alternatively reset `botInitialized = false` in the catch so the next `ready` (or a retry timer) re-runs init. One-line-ish in both bots.

**2. Jester cards/stones: full-array read-modify-write with cache-sourced reads → dupe/loss on concurrency, and a full-collection-wipe path** — Severity: HIGH, Effort: S
- Evidence: `LunaJesterMain\util\managers\cards.ts:112-119` — `removeCard` = `getUserCards()` (cached, and its catch at `cards.ts:102-104` returns `[]` on ANY error) → JS filter → `$set { cards: updatedCards }` with `upsert: true`. Same in `stones.ts:73-118` (`addStone`/`removeStone` `$set { stones }` from a possibly-stale cached array). Butler mirrors it in `LunaButlerMain\util\managers\card_manager.ts` / `stone_manager.ts`.
- Failure scenario: (a) corrupt legacy `data` string or transient read error → `getUserCards` returns `[]` → `removeCard` writes `cards: []` + `$unset data` → user's entire card collection silently destroyed, unrecoverable. (b) concurrent trade-sell + chest/pull on the same user → one write clobbers the other → sold card resurrected (dupe exploit on tradeable assets) or new stone lost. This is exactly the class the money fixes closed, unfixed for cards/stones.
- Fix: use `$pull: { cards: { id: cardId } }` / `$push` — the correct pattern already exists in the same repo at `LunaJesterMain\util\managers\inventory.ts:47-50` (`removeItem` uses `$pull`) and `inventory.ts:36-40` (`$push`). Never `$set` a whole array whose source read can fail to `[]`; make `getUserCards` rethrow instead of returning `[]` when used inside write paths.

**3. Butler ProfilesManager: whole-doc `$set` from a 15s cache on a collection Jester writes with `$inc` → cross-bot lost updates** — Severity: HIGH (silent stat corruption, historically already happened), Effort: S
- Evidence: `LunaButlerMain\util\managers\profiles.ts:203-214` — `incrementField` = `getProfile()` (15s cache, `profiles.ts:139-141`) → mutate → `setProfile` `$set: { data: profile }` (whole object, `profiles.ts:194-198`). Called on EVERY message (`index.ts:726` `addMessage`) and every voice tick (`voice_tracker.ts:140,208`). Jester writes the same `profiles` docs atomically with `$inc: { 'data.field' }` (`LunaJesterMain\util\managers\profiles.ts:171-179`).
- Failure scenario: user chats in Butler-tracked channels while finishing a Jester game → Butler holds a profile cached up to 15s, then `$set`s the entire `data` object → Jester's `luna_fantasy.wins`/`money_earned` increments written in between are erased. No error anywhere. The `b`-field recovery code at `profiles.ts:150-167` ("data still partially zeroed by Jester's old bug") is a scar from this exact class.
- Fix: rewrite Butler `incrementField` to `$inc: { ['data.' + field]: amount }` with upsert — copy Jester's `profiles.ts:171-179` verbatim. Keep `setProfile` only for explicit full-profile admin writes.

**4. Butler LevelManager.addXP: read-modify-write + 60s cache + double sequential `$set` → lost XP under normal use** — Severity: MEDIUM, Effort: M
- Evidence: `LunaButlerMain\util\managers\levels.ts:107-131` — `addXP` reads `getUserData` (60s cache, `levels.ts:88-94`), mutates, `$set`s all four fields (`levels.ts:99-105`). `addTextXP` then does a SECOND read+`$set` for `messages` (`levels.ts:151-155`); `addVoiceTime` same for `voiceTime` (`levels.ts:168-172`).
- Failure scenario: user is in voice while chatting (very common) → text XP write and voice XP write interleave, both based on stale reads → one overwrite loses the other's XP/counters. Silent, cumulative, users notice missing XP over months.
- Fix: single `findOneAndUpdate` with `$inc: { xp, messages/voiceTime }` and `returnDocument: 'after'`, then compute `level` from the returned xp and `$set` level only when changed — same atomic pattern as Jester `game_wins.ts:64-72` and Butler's own `tryClaimLeaderboardUpdateLevels` (`levels.ts:330-351`).

**5. Butler InventoryManager.addItem: `$set items` from an error-swallowed read → inventory wipe on a transient blip during purchase** — Severity: MEDIUM-HIGH, Effort: S
- Evidence: `LunaButlerMain\util\managers\inventory.ts:69-98` — `addItem` calls `getUserInventory` whose catch (`inventory.ts:56-59`) returns `[]` on any error, then `$set: { items: updatedItems }` + `$unset data` with upsert.
- Failure scenario: Atlas hiccup or corrupt legacy doc exactly when a user buys a background → read yields `[]` → write replaces the whole inventory with the one new item. All previously purchased backgrounds/roles gone, no error shown.
- Fix: `$push` with a duplicate-guard filter (`{ _id: userId, 'items.id': { $ne: item.id } }`) and `$pull` for `removeItem` — again the Jester `inventory.ts` pattern. Make read errors in write paths rethrow instead of returning `[]`.

**6. Seluna limited-stock: whole-doc read-modify-write of `shop_stocks` → oversell + cross-shop clobber** — Severity: MEDIUM, Effort: S-M
- Evidence: `LunaJesterMain\commands\shops\seluna_vendor.ts:558-629` — buy flow: `selunaGet('shop_stocks')` (whole doc, all channels) → check `remainingStock` in JS → mutate → `selunaSet('shop_stocks', allStocks)` (whole doc, `util\helpers\seluna_helpers.ts:16-22`).
- Failure scenario: two buyers race for the last limited item → both read `1`, both pass the check at line 570, both write `0` → item oversold beyond its cap (Seluna trades only rare items — visibility is high when it breaks). Also any concurrent stock write for the dev shop clobbers the production channel's stock since all channels live in one doc.
- Fix: store per-shop stock as its own doc and decrement with a guarded atomic update: `updateOne({ id: 'stock_'+shopKey, [itemId]: { $gt: 0 } }, { $inc: { [itemId]: -1 } })`; treat `modifiedCount === 0` as sold-out. Same conditional-update idea as the already-hardened points deducts.

**7. Jester connection timeouts oversized: 60s serverSelection / 60s waitQueue / 90s socket → minute-long full-bot freezes during Atlas blips** — Severity: MEDIUM, Effort: S
- Evidence: `LunaJesterMain\util\infra\database_helper.ts:8-22` (`serverSelectionTimeoutMS: 60000`, `waitQueueTimeoutMS: 60000`, `socketTimeoutMS: 90000`, `connectTimeoutMS: 60000`). Butler and Oracle both use 5s/30s (`LunaButlerMain\util\helpers\database_helper.ts:8-16`, `LunaOracle\util\helpers\database_helper.ts:6-14`). Managers hit collections directly via `getMongoCollection(...)!` (e.g. `game_wins.ts:24-30`) — the `withConnection` retry wrapper is only used by config readers (`config_reader.ts`, `game_config.ts`, `vendor_config_db.ts`, `webhook_sync.ts`).
- Failure scenario: Atlas 30-second blip → every in-flight Jester DB call hangs up to 60-90s instead of failing fast at ~5s. The interaction queue's concurrency slots (GAME:15, SHOP:8) fill with hung ops → ALL games/shops appear frozen; users spam buttons; Discord interactions expire (3s ack / 15min token) → wave of "interaction failed". Note `connectionHealth.getBlockedReason()` (`util\infra\connection_health.ts:177-181`) is Discord-gateway-latency only — it does NOT gate on DB health, so games happily start while the DB is down and then hang mid-game.
- Fix: align Jester's `MONGO_OPTIONS` with Butler's profile (`serverSelectionTimeoutMS: 5000`, `connectTimeoutMS: 5000`, `socketTimeoutMS: 30000`, drop waitQueue to ~10s). Optional cheap add: an `isMongoConnected()` check (already exported, `database_helper.ts:144-146`) inside the pre-game gate next to `getBlockedReason()`.

**8. Systemic: `JSON.parse(doc.data)` on legacy-format docs without try/catch → one corrupt doc poisons game-end flows repeatedly** — Severity: MEDIUM-LOW (but pairs badly with #2/#5), Effort: S
- Evidence (unguarded sites): Jester `game_wins.ts:40,57` (readWins/readNemesis — called in `addNemesisWin`, i.e. the game-end pattern), `profiles.ts:101,132`, `inventory.ts:28,74`, `cards.ts:43` (inside try but rethrows), `stones.ts:63`; Butler `levels.ts:68,290`, `chat_tracker.ts:100`. Guarded correctly in Butler `profiles.ts:125` (try/catch) and `seluna_helpers.ts:10-12` — the safe pattern already exists in-repo.
- Failure scenario: one document with a truncated/corrupt `data` string (e.g. from the old st.db era) → every game that user finishes throws in `addNemesisWin`/`getProfile` → post-game stat writes fail forever for that user; combined with the `catch → []` paths it can escalate to wipes (#2, #5).
- Fix: a tiny `safeParseData(doc)` helper per repo (try/catch, return `{}`/`[]` + `console.error` with the `_id` so corrupt docs get noticed) and swap the ~10 call sites. Do NOT return empty into `$set`-whole-array write paths (see #2/#5 fixes).

DELIBERATELY DOWN-RANKED (would be over-engineering at 1,400 users — live data says so):
- **Missing secondary indexes**: the "ZERO indexes" claim is outdated — 117 indexes exist; Butler's properties/eclipse/fence subsystems call `ensureIndexes()` (`properties.ts:63-66`, `eclipse.ts:51-52`, `fence_manager.ts:92`). The scariest-looking query — `findNemesis`'s unanchorable `$regex: '_userId$'` collscan (`LunaJesterMain\util\managers\game_wins.ts:152-155`) — runs against 62 documents. `game_wins.findOne({ id })` collscans 68 docs. Leaderboards `find({}).toArray()` + JS sort (`LunaButlerMain\util\managers\points.ts:303`, `levels.ts:252`) touch a few thousand tiny docs. All fine today. Only if the community 10x's: add `{ id: 1 }` on `game_wins` and restructure nemesis docs with a `players: [a,b]` indexed array.
- **Unbounded growth**: 6,137 lunari_transactions in ~4 months (~1.5k/month), whole DB 22 MB — years of headroom even on Atlas M0. Cheap insurance when convenient: one TTL index (`createdAt`, e.g. 180-365 days) on the three `*_transactions` collections — but coordinate retention with the dashboard first (it reads these). Jester's `active_game_states` self-cleans stale (>15 min) docs on restore (`game_state_persistence.ts:109-120`) and the daily restart keeps it near-empty — no action needed.
- Minor notes, not worth their own findings: Oracle opens a redundant second MongoClient just for profile polling (`LunaOracle\index.ts:365`, silently disabled forever if its first connect fails); Jester `gameStatePersistence.flushAll()` (`game_state_persistence.ts:138-147`) logs "Flushed pending saves" but actually discards them (shutdown paths use `saveGameStateImmediate`, so cosmetic); Butler `chat_tracker.flushPendingMessages` (`chat_tracker.ts:169-197`) can double-count a batch if the flush throws mid-loop (counts already written aren't cleared) — rare, low stakes.

ALREADY SOLID — do not touch:
- Butler/Oracle `MONGO_OPTIONS`: fail-fast 5s serverSelection, `retryWrites`/`retryReads`, single shared pool per process (`LunaButlerMain\util\helpers\database_helper.ts:8-16`, Oracle mirror).
- Jester connection layer resilience: retry-with-backoff + recoverable-error classification (`database_helper.ts:32-97`), `withConnection` one-shot reconnect+retry, non-blocking `initializeMongoPool` at startup (bot serves Discord while DB reconnects — no zombie problem, unlike Butler/Oracle), `topology.isConnected()` fast-path verified valid on driver v7.
- Jester ProfilesManager: fully atomic `$inc` dot-path writes (`profiles.ts:171-193`) — the reference implementation for findings 3-4.
- Jester GameWinsManager.addWin / setWins: atomic `findOneAndUpdate` + `$inc` + `$setOnInsert` (`game_wins.ts:64-72`).
- Jester InventoryManager: `$push`/`$pull` array ops (`inventory.ts:36-53`) — the reference for finding 2/5.
- Consistent `$unset: { data: "" }` legacy-format cleanup on every write path, all three bots — the st.db migration hygiene is done right.
- Atomic leaderboard-update claim pattern (`tryClaimLeaderboardUpdate` in Butler `points.ts` and `levels.ts:330-351`) — correct race prevention.
- Jester game state persistence: debounced saves, error-swallowing so persistence never breaks game logic, stale-state cleanup on restore.
- Butler manager collection getters throw a clear `"[X] MongoDB not connected"` error instead of a null TypeError (`levels.ts:20-30`, `inventory.ts:23-27`, `points.ts:35-38`).
- Fire-and-forget transaction logging (`webhook_sync.ts` both bots) — never blocks bot ops, correct choice.
